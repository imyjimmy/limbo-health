# MySQL to PostgreSQL Migration Plan

Status: Draft
Owner: Limbo Health
Last updated: 2026-03-21

## Goal

Migrate the shared `auth-api` + `scheduler-api` data plane from MySQL to the existing PostgreSQL stack with the lowest-risk path that still gets the repo to the lean end state described in `docs/specs/aws-railway-replacement-spec.md`:

- one PostgreSQL database
- no long-term MySQL dependency
- minimal application churn
- a rehearsable cutover with a short maintenance window

This plan is intentionally concrete to the current repo, not a generic MySQL/Postgres checklist.

## What The Repo Tells Us

### Current database split

- `auth-api` uses MySQL via `mysql2/promise` in `apps/auth-api/server.js`.
- `scheduler-api` uses MySQL via `mysql2/promise` in `apps/scheduler-api/config/database.js`.
- `records-workflow-api` already uses Postgres via `pg` in `apps/records-workflow-api/src/db.js`.

### Shared schema reality

`auth-api` and `scheduler-api` are already effectively sharing one MySQL schema defined in `apps/scheduler-api/db/schema.sql`.

The active table set is:

- `roles`
- `users`
- `oauth_connections`
- `user_settings`
- `repositories`
- `repository_access`
- `scan_sessions`
- `provider_profiles`
- `service_categories`
- `services`
- `services_providers`
- `appointments`
- `blocked_periods`
- `invoices`
- `secretaries_providers`
- `consents`
- `settings`
- `webhooks`
- `migrations`

### Single Postgres database is structurally viable

There are no table-name collisions between `apps/scheduler-api/db/schema.sql` and `apps/records-workflow-api/db/schema.sql`, so the simplest final shape is:

- keep one Postgres instance
- keep one Postgres database
- add the migrated auth/scheduler tables alongside the existing records-workflow tables

For the first cut, the lowest-risk option is to keep using the existing records-workflow Postgres database name and add the new tables there. Renaming the database can happen later if desired.

## Recommended Migration Strategy

### Summary

Use a three-part migration:

1. Add a shared Postgres adapter that preserves the current `mysql2` calling style as much as possible.
2. Create a Postgres-native schema for the current MySQL tables.
3. Rehearse and execute a snapshot-based cutover with a brief write freeze for `auth-api` and `scheduler-api`.

This is not a true zero-downtime migration. For this repo, the safest first version is a short maintenance window rather than dual-write or CDC.

### Why this approach

The codebase is not dominated by complex SQL, but it does depend on:

- `insertId`
- `affectedRows`
- `pool.getConnection()`
- `beginTransaction()` / `commit()` / `rollback()`
- `?` placeholders
- MySQL-only SQL forms like `ON DUPLICATE KEY UPDATE` and `DATE_SUB(...)`

If we swap the driver and every route at once, the blast radius is larger than necessary. A thin compatibility layer lets us keep most call sites stable while rewriting only the incompatible SQL.

## Target PostgreSQL Schema Decisions

### Database-level decisions

- Reuse the existing Postgres instance already serving `records-workflow-api`.
- Add auth/scheduler tables to the same database.
- Keep all migrated tables in `public` for the first cut to avoid schema/search-path churn.
- Enable `citext` once at bootstrap time.

### Type mapping decisions

| MySQL pattern | PostgreSQL target | Notes |
| --- | --- | --- |
| `int AUTO_INCREMENT` | `integer generated by default as identity` | Preserve existing IDs during import, then reset sequences with `setval`. |
| `varchar`, `text`, `longtext` | `text` or `varchar(n)` | Prefer `text` unless length has app meaning. |
| `datetime` | `timestamp without time zone` | Preserve current app behavior first. The code currently writes naive timestamps and compares against DB `NOW()`. |
| `date` | `date` | Direct mapping. |
| `timestamp` | `timestamp without time zone` | Keep semantics aligned with current MySQL usage on first cut. |
| `tinyint` boolean flags | `boolean` where truly boolean, otherwise `smallint` or `integer` | Keep role permission counters as integers. |
| MySQL `JSON` | `jsonb` | Applies to `provider_profiles.languages`, `board_certifications`, and `working_plan`. |
| MySQL `ENUM('admin', 'read-write', 'read-only')` | `text check (...)` | Avoids Postgres enum migration friction. |
| `utf8mb4_unicode_ci` case-insensitive comparisons | `citext` or explicit lower-case indexes | Needed to preserve MySQL-like equality behavior for email and username. |

### Columns that should become `citext`

Use `citext` for:

- `users.email`
- `provider_profiles.username`
- `oauth_connections.provider_email`

This preserves the case-insensitive behavior the app currently gets from MySQL collations for email and username lookups.

### Columns that should stay integer-like

Do not convert these to booleans:

- `roles.appointments`
- `roles.customers`
- `roles.services`
- `roles.users`
- `roles.system_settings`
- `roles.user_settings`
- `roles.webhooks`
- `roles.blocked_periods`

The seed data uses `0` and `15`, so these are permission values, not true booleans.

### Columns that should become booleans

Use `boolean` for:

- `roles.is_admin`
- `users.is_private`
- `user_settings.notifications`
- `user_settings.google_sync`
- `user_settings.caldav_sync`
- `scan_sessions.is_revoked`
- `services.is_private`
- `appointments.is_unavailability`
- `webhooks.is_ssl_verified`

### Table-specific notes

- `repository_access.access_level`
  - Use `text not null check (access_level in ('admin', 'read-write', 'read-only'))`.
- `provider_profiles`
  - Store JSON-like columns as `jsonb`.
  - Keep `user_id` as both PK and FK.
- `settings.value`
  - Keep as `text`, not `jsonb`, because the table mixes JSON strings and plain string settings.
- `migrations`
  - Preserve the table if any legacy code depends on it, but it is low-risk and not on the hot path.

## Application Change Plan

### Phase 1: Add a shared Postgres compatibility adapter

Add a shared adapter module used by both `auth-api` and `scheduler-api`.

Recommended surface:

- `createPool({ connectionString })`
- `pool.query(sql, params)`
- `pool.execute(sql, params)`
- `pool.getConnection()`
- `connection.query(sql, params)`
- `connection.execute(sql, params)`
- `connection.beginTransaction()`
- `connection.commit()`
- `connection.rollback()`
- `connection.release()`

The adapter should:

- translate `?` placeholders to `$1`, `$2`, `$3`, and so on
- return `[rows]` for `SELECT`
- return a mysql2-like result object for `INSERT` / `UPDATE` / `DELETE`
- expose `affectedRows`
- expose `insertId` when the SQL includes `RETURNING id`

This reduces code churn, but it does not eliminate the need to rewrite MySQL-only SQL syntax.

### Phase 2: Rewrite the known incompatible SQL

### `apps/auth-api/server.js`

Needs:

- replace insert statements that rely on `insertId` with `RETURNING id`
- replace the repository-access merge query:
  - from `ON DUPLICATE KEY UPDATE access_level = VALUES(access_level)`
  - to `ON CONFLICT (repo_id, user_id) DO UPDATE SET access_level = EXCLUDED.access_level`

### `apps/auth-api/routes/repos.js`

Needs:

- `repositories` upsert rewrite:
  - `ON DUPLICATE KEY UPDATE updated_at = NOW()`
  - becomes `ON CONFLICT (id) DO UPDATE SET updated_at = NOW()`
- `repository_access` upsert rewrite:
  - `ON DUPLICATE KEY UPDATE access_level = 'admin'`
  - becomes `ON CONFLICT (repo_id, user_id) DO UPDATE SET access_level = 'admin'`

### `apps/auth-api/routes/scan.js`

Needs:

- replace MySQL interval syntax:
  - `NOW() - INTERVAL 15 MINUTE`
  - `NOW() - INTERVAL 5 MINUTE`
- with Postgres interval syntax:
  - `NOW() - INTERVAL '15 minutes'`
  - `NOW() - INTERVAL '5 minutes'`

### `apps/scheduler-api/routes/appointments.js`

Needs:

- replace MySQL double-quoted string literals:
  - `slug = "customer"`
  - with:
  - `slug = 'customer'`
- replace `insertId`-based inserts with `RETURNING id`

### `apps/scheduler-api/routes/google-auth.js`

Needs:

- replace user inserts that expect `insertId` with `RETURNING id`
- keep `NOW()` calls, which are valid in Postgres

### `apps/scheduler-api/routes/providers.js`

Needs:

- replace two upserts on `provider_profiles`
  - use `ON CONFLICT (user_id) DO UPDATE SET ...`
- replace insert paths that read `insertId` from new `users` rows with `RETURNING id`

### `apps/scheduler-api/routes/admin.js`

Needs:

- replace `users` insert returning `insertId` with `RETURNING id`
- replace `services` insert returning `insertId` with `RETURNING id`

### `apps/scheduler-api/routes/billing.js`

Needs:

- replace `DATE_SUB(NOW(), INTERVAL 30 DAY)` with `NOW() - INTERVAL '30 days'`
- replace invoice inserts that depend on `insertId` with `RETURNING id`

### Files that mostly work once the adapter exists

These files appear to need little or no SQL syntax change beyond driver migration:

- `apps/scheduler-api/routes/user-keys.js`
- `apps/scheduler-api/routes/webrtc.js`
- `apps/scheduler-api/routes/providers.js` read-only queries
- `apps/scheduler-api/middleware/auth.js`
- `apps/scheduler-api/utils/auth.ts`

### Phase 3: Move environment/config to Postgres

### App config

Introduce `DATABASE_URL` for both `auth-api` and `scheduler-api`.

Keep a temporary fallback during rollout:

- if `DATABASE_URL` is set, use Postgres
- otherwise fall back to current MySQL env vars

That gives staging and cutover flexibility.

### Infra/config files that will need updates

- `docker-compose.yml`
- `docker-compose.production.yml`
- `deploy/aws/lean/docker-compose.ec2.yml`
- `scripts/backup-mysql.sh`
- `deploy/aws/lean/backup-databases-to-s3.sh`
- any helper script or README that still assumes `mysqldump` or a `mysql` service

### Test updates

`apps/auth-api/__tests__/auth-me.test.ts` currently shells out to the `mysql` CLI directly.

Replace that with either:

- a small `pg` helper used by the test, or
- a shell call to `psql`

Using `pg` from the test suite is preferred because it avoids shell and client binary assumptions.

## Data Migration Plan

### Phase 4: Build the Postgres schema first

Create a new Postgres DDL file for the auth/scheduler tables, for example:

- `apps/core-db/schema.postgres.sql`

Requirements:

- create `citext` extension
- create all auth/scheduler tables
- preserve primary keys and foreign keys
- replace MySQL enum and collation behavior as described above
- seed `roles`, `settings`, and `migrations`

Do not attempt to auto-convert the MySQL DDL at deploy time. Check in a hand-authored Postgres schema.

### Phase 5: Write a repo-owned migration script

Add a one-shot migration script, for example:

- `scripts/migrate-core-mysql-to-postgres.ts`

The script should:

1. Connect to source MySQL and target Postgres.
2. Truncate or clear only the migrated auth/scheduler tables in Postgres.
3. Copy rows table-by-table in foreign-key order.
4. Preserve source IDs exactly.
5. Normalize booleans and JSON columns.
6. Reset Postgres sequences after each identity-backed table import.
7. Print row counts before and after.
8. Exit non-zero on any mismatch.

### Recommended import order

Import in this order:

1. `roles`
2. `users`
3. `oauth_connections`
4. `user_settings`
5. `repositories`
6. `repository_access`
7. `scan_sessions`
8. `provider_profiles`
9. `service_categories`
10. `services`
11. `services_providers`
12. `appointments`
13. `blocked_periods`
14. `invoices`
15. `secretaries_providers`
16. `consents`
17. `settings`
18. `webhooks`
19. `migrations`

### Data normalization rules

- Preserve all integer IDs exactly.
- Convert `tinyint` booleans to true/false.
- Convert MySQL JSON columns to `jsonb`.
- Keep datetime-like values as `timestamp without time zone`.
- Preserve nulls exactly.

### Sequence reset rule

After importing each identity-backed table, run the Postgres equivalent of:

```sql
SELECT setval(
  pg_get_serial_sequence('users', 'id'),
  COALESCE((SELECT MAX(id) FROM users), 1),
  true
);
```

Run this for every imported integer PK table.

## Cutover Plan

### Preferred rollout

1. Land the Postgres schema, adapter, and query rewrites behind config.
2. Stand up a staging environment where `auth-api` and `scheduler-api` run on Postgres only.
3. Rehearse the migration from a recent MySQL dump into staging.
4. Run app smoke tests against staging.
5. Schedule a short maintenance window for production write freeze.
6. Stop or disable write paths for `auth-api` and `scheduler-api`.
7. Take a final MySQL snapshot/dump.
8. Run the migration script against the final snapshot.
9. Point `auth-api` and `scheduler-api` at Postgres.
10. Run smoke tests immediately.
11. Keep the old MySQL data volume and dump intact until the soak period ends.

### Why not dual-write first

Dual-write would reduce downtime, but it adds failure modes the repo does not currently have:

- partial-write divergence
- replay/idempotency complexity
- cross-store transaction ambiguity

The codebase is small enough that a short maintenance window is a better first migration.

## Verification Checklist

Verify all of the following after each rehearsal and again during cutover:

### Row-level validation

- row counts match for every migrated table
- max primary key matches for every identity-backed table
- no orphaned foreign keys exist
- unique indexes are still satisfiable

### Behavior validation

- Nostr auth login works
- Google auth login works for both `auth-api` and `scheduler-api`
- `/api/auth/me` returns the same connections and profile data
- repo registration and access checks work
- scan session create, revoke, and cleanup work
- provider profile create/update works
- provider profile picture upload still updates DB correctly
- appointment booking works
- admin appointment listing works
- invoice creation works
- billing stats work
- WebRTC room join logic still resolves appointments correctly

### Specific compatibility checks

- email lookup remains case-insensitive
- provider username lookup remains case-insensitive
- `repository_access` upserts preserve the expected access level
- timestamps still sort and compare correctly

## Rollback Plan

If smoke tests fail after switching to Postgres:

1. Stop `auth-api` and `scheduler-api`.
2. Restore the previous environment variables pointing them back to MySQL.
3. Restart the apps on MySQL.
4. Keep Postgres data for inspection, but treat MySQL as source of truth until the issue is fixed.

Do not destroy the MySQL volume or final dump until:

- production has soaked successfully on Postgres
- backups are verified
- at least one restore rehearsal has succeeded

## Recommended Implementation Order In This Repo

1. Add the checked-in Postgres DDL for auth/scheduler tables.
2. Add the shared Postgres compatibility adapter.
3. Switch `auth-api` to the adapter.
4. Rewrite auth-specific upserts and insert-return paths.
5. Switch `scheduler-api` to the adapter.
6. Rewrite scheduler-specific upserts, interval syntax, and insert-return paths.
7. Add the MySQL-to-Postgres migration script.
8. Update local Docker Compose to run auth/scheduler against Postgres.
9. Update tests to seed Postgres instead of MySQL.
10. Rehearse on staging with a recent dump.
11. Update AWS compose and backup scripts.
12. Execute production cutover.

## Open Questions Before Production Cutover

These do not block implementation, but they do block calling the migration "seamless":

- What is the acceptable production maintenance window?
- Do we want to keep the current records-workflow database name for the first cut, or rename to `limbo_health` later?
- Are any MySQL JSON columns currently storing malformed or double-encoded JSON?
- Are there production rows that depend on MySQL's case-insensitive comparisons in ways not covered by `citext`?
- Do we need to preserve any legacy EasyAppointments behavior tied to `migrations`, `webhooks`, `consents`, or `blocked_periods`?

## Bottom Line

This migration is feasible with the current repo.

The cleanest path is:

- one checked-in Postgres schema
- one compatibility adapter
- a small set of explicit SQL rewrites
- one scripted snapshot migration
- one rehearsed maintenance-window cutover

That is enough to get off MySQL without turning the migration itself into a new platform project.
