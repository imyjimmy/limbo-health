# NOSTR Encrypted Object Storage Service Spec (S3 Backend)

## 1. Purpose
Provide paid, multi-tenant object storage for NOSTR users where:
- Users authenticate with NOSTR keys.
- Users are responsible for client-side encryption.
- Service stores ciphertext only and never requires plaintext keys.

This service is intentionally separate from Limbo Health.

## 2. Non-Goals
- No server-side content processing of plaintext files.
- No server-managed end-user encryption keys.
- No guarantee that arbitrary bytes are cryptographically strong encryption.

## 3. Trust Model
- Trusted: service control plane (auth, upload broker, billing), AWS account, IAM roles.
- Untrusted: upload clients and object payloads.
- Sensitive assumption: if client bypasses encryption library, they may upload plaintext-like bytes unless blocked by protocol checks.

## 4. High-Level Architecture
- API service (stateless): NOSTR auth verification, upload/download authorization, quota checks, billing metering.
- Metadata DB (Postgres or DynamoDB): object manifests, tenant limits, usage counters, soft-delete state.
- S3 bucket: blob storage only (`objects/<pubkey>/<object_id>`).
- Optional background workers: lifecycle cleanup, malware scanning of ciphertext metadata only, usage aggregation.
- CloudWatch + CloudTrail: audit and operational telemetry.

## 5. Tenant Identity and Namespacing
- Tenant ID: normalized hex NOSTR pubkey.
- Namespace prefix: `objects/<tenant_pubkey>/`.
- Object ID: UUIDv7 or 128-bit random string generated by server.
- Never trust client-supplied key paths directly.

## 6. Encryption Contract (Fail-Closed)
The service enforces container conformance, not decryption.

### 6.1 Required Container Format
Every upload must include:
- Magic bytes: `NSENC1`
- Version: `1`
- Algorithm ID (allowlist): `XCHACHA20_POLY1305` or `AES256_GCM`
- Nonce length and nonce bytes
- Ciphertext length and bytes
- Authentication tag
- Manifest hash/checksum (SHA-256) supplied at init phase

### 6.2 Enforcement Rule
Only objects whose first bytes parse as the required container header are accepted as "active" objects.

### 6.3 Limitation
Container conformance reduces accidental plaintext uploads but cannot prove payload quality against a malicious client.

## 7. AuthN/AuthZ (NOSTR)

### 7.1 NOSTR Login
1. `POST /v1/auth/challenge` returns nonce + expiration.
2. Client signs challenge as NOSTR event.
3. `POST /v1/auth/verify` validates signature and issues JWT/session.

### 7.2 Authorization Model
- JWT subject = tenant pubkey.
- All object operations restricted to `objects/<subject_pubkey>/`.
- Service role signs short-lived pre-signed S3 URLs; clients never receive AWS credentials.

## 8. API Contract (v1)

### 8.1 `POST /v1/objects/init-upload`
Purpose: reserve object ID and issue constrained pre-signed PUT URL.

Request:
- `container_header_b64`
- `content_length`
- `sha256`
- `declared_algorithm`
- `metadata` (optional encrypted filename hint)

Validation:
- JWT valid and tenant active.
- Header parses and algorithm allowlisted.
- Size <= tenant max object size.
- Tenant quota not exceeded.

Response:
- `object_id`
- `s3_key`
- `put_url` (TTL 5-10 min)
- required headers (`x-amz-checksum-sha256`, `content-length`, metadata tags)

### 8.2 `POST /v1/objects/complete-upload`
Purpose: finalize upload after S3 PUT.

Request:
- `object_id`
- `etag`
- `size`
- `sha256`

Validation:
- Object exists in `PENDING_UPLOAD`.
- S3 HeadObject matches expected size/checksum.
- Header bytes still parse as valid container format.

Result:
- Mark object `ACTIVE`.
- Update tenant usage counters.
- Emit audit event.

Failure handling:
- Mark `REJECTED` and delete object (or quarantine prefix) on mismatch.

### 8.3 `POST /v1/objects/get-download-url`
Purpose: issue short-lived GET URL for owner.

Request:
- `object_id`

Validation:
- Caller owns object.
- Object status `ACTIVE`.

Response:
- `get_url` (TTL 1-5 min)

### 8.4 `POST /v1/objects/delete`
Purpose: logical or hard delete.

Behavior:
- Mark metadata `DELETED` and enqueue S3 delete.
- Idempotent endpoint.

### 8.5 `GET /v1/objects/list`
Purpose: list object metadata only (no plaintext metadata requirement).

## 9. Object State Machine
- `PENDING_UPLOAD` -> `ACTIVE`
- `PENDING_UPLOAD` -> `REJECTED`
- `ACTIVE` -> `DELETED`
- `REJECTED`/`DELETED` terminal for client access

## 10. Metadata Model

### 10.1 `tenants`
- `pubkey` (PK)
- `plan_id`
- `storage_quota_bytes`
- `max_object_size_bytes`
- `status` (`ACTIVE`, `SUSPENDED`)
- `created_at`, `updated_at`

### 10.2 `objects`
- `object_id` (PK)
- `pubkey` (indexed)
- `s3_bucket`
- `s3_key`
- `size_bytes`
- `sha256`
- `algorithm`
- `container_version`
- `status`
- `created_at`, `completed_at`, `deleted_at`

### 10.3 `usage_ledger`
- `pubkey`
- `period_start`
- `stored_bytes`
- `egress_bytes`
- `request_count`

## 11. AWS S3 and IAM Requirements

### 11.1 Bucket Configuration
- Block Public Access: all enabled.
- Object Ownership: bucket owner enforced.
- Versioning: enabled.
- Default encryption: SSE-KMS (or SSE-S3 minimum).
- Lifecycle:
  - Abort incomplete multipart uploads (7 days).
  - Expire `REJECTED`/quarantine objects quickly.
- Server access logging or CloudTrail data events enabled.

### 11.2 Access Pattern
- Only service IAM role can create pre-signed URLs.
- Clients interact with S3 only via pre-signed URLs.
- No public bucket ACLs or policies.

### 11.3 Bucket Policy Baseline (Intent)
Deny when:
- Request is not TLS (`aws:SecureTransport=false`).
- Principal is outside allowed service roles.
- Required checksum header is missing on PUT.

Allow:
- `s3:PutObject`, `s3:GetObject`, `s3:AbortMultipartUpload` only through brokered flow.

## 12. Quotas, Billing, Abuse
- Per-tenant hard quota in bytes.
- Per-tenant max object size.
- Per-tenant request rate limits on init/complete/download URL endpoints.
- Soft and hard suspension modes:
  - Soft: read allowed, write blocked.
  - Hard: both blocked except delete/export path.
- Budget alarms on AWS account and per-tenant anomaly detection.

## 13. Observability and Audit
- Log all auth, init-upload, complete-upload, delete, and presign actions with request IDs.
- Emit metrics:
  - upload_init_count, upload_complete_success/fail
  - rejected_container_count
  - active_storage_bytes_by_tenant
  - presign_latency_ms
- Enable CloudTrail S3 data events for forensic traceability.

## 14. Security Controls Checklist
- Least-privilege IAM roles.
- Rotate signing secrets and JWT keys.
- WAF/rate limiting on public API.
- Strict JSON schema validation for all requests.
- Replay protection for NOSTR challenge nonces.
- Optional quarantine prefix for suspicious uploads.

## 15. Compliance and Data Handling Notes
- Service stores ciphertext and minimal operational metadata.
- Metadata (size, timestamps, access patterns) is still sensitive and should be protected.
- Publish clear customer terms: user is solely responsible for key custody and decryptability.

## 16. SLO and Failure Modes
- API availability target: 99.9% monthly.
- Upload finalization p95 < 500 ms excluding object transfer time.
- If complete step fails after successful PUT, object remains inaccessible until reconciler resolves it.

## 17. MVP Scope (Phase 1)
- NOSTR auth challenge/verify.
- init-upload + complete-upload + download-url + delete + list.
- Single S3 bucket in one region.
- Quotas, rate limits, usage metering, and basic billing export.

## 18. Future Enhancements (Phase 2+)
- Multi-region replication with tenant-selectable residency.
- Client SDKs for automatic encryption container generation.
- Encrypted metadata index (search over encrypted filename tags).
- Account recovery mechanisms outside service key custody.

## 19. Open Decisions
- Postgres vs DynamoDB for metadata plane.
- Exact container binary format (custom vs JOSE/age wrapper).
- Quarantine vs immediate delete behavior on validation failure.
- Billing model: provisioned GB vs pay-as-you-go.

```json

  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyInsecureTransport",
      "Effect": "Deny",
      "Principal": "*",
      "Action": "s3:*",
      "Resource": [
        "arn:aws:s3:::<BUCKET_NAME>",
        "arn:aws:s3:::<BUCKET_NAME>/*"
      ],
      "Condition": {
        "Bool": {
          "aws:SecureTransport": "false"
        }
      }
    },
    {
      "Sid": "DenyPutWithoutChecksum",
      "Effect": "Deny",
      "Principal": "*",
      "Action": "s3:PutObject",
      "Resource": "arn:aws:s3:::<BUCKET_NAME>/objects/*",
      "Condition": {
        "Null": {
          "s3:x-amz-checksum-sha256": "true"
        }
      }
    },
    {
      "Sid": "AllowOnlyBrokerRoleObjectOps",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::<ACCOUNT_ID>:role/<ROLE_NAME>"

        "s3:PutObject",

      ],
```
